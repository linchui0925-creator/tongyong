# TongYong Agent 会话记录存储架构详细文档

## 一、当前数据库架构

### 1.1 核心设计决策：共用表 vs 独立表

| 方案 | 当前实现 | 说明 |
|------|----------|------|
| **共用数据表** | ✅ 采用 | 所有会话的消息存储在同一个 `messages` 表，通过 `session_id` 字段隔离 |
| 独立数据表 | ❌ 未采用 | 每个会话拥有独立的数据库表 |

### 1.2 技术原因

选择**共用数据表 + session_id 隔离**的原因：

| 因素 | 说明 |
|------|------|
| **扩展性** | 支持任意数量的会话，无需动态创建/删除表 |
| **事务一致性** | 跨会话操作可保证原子性 |
| **维护简单** | 单一数据表便于备份、迁移、监控 |
| **SQLite 优化** | SQLite 对小表（<100万行）性能优异 |

## 二、表结构定义

### 2.1 会话表 (sessions)

```sql
CREATE TABLE sessions (
    id TEXT PRIMARY KEY,           -- UUID，会话唯一标识
    name TEXT NOT NULL,             -- 会话名称（可读性）
    created_at TEXT NOT NULL,         -- 创建时间 ISO8601
    updated_at TEXT NOT NULL         -- 更新时间 ISO8601
);
```

**字段设计说明**：

| 字段 | 类型 | 约束 | 设计原因 |
|------|------|------|----------|
| id | TEXT | PRIMARY KEY | 使用 UUID 避免分布式冲突，支持跨设备同步 |
| name | TEXT | NOT NULL | 用户可读性，便于识别会话 |
| created_at | TEXT | NOT NULL | 时间线排序，支持历史追溯 |
| updated_at | TEXT | NOT NULL | 用于最近对话排序，显示会话列表顺序 |

**当前索引**：
- 主键索引：`sqlite_autoindex_sessions_1`（自动生成）

---

### 2.2 消息表 (messages)

```sql
CREATE TABLE messages (
    id INTEGER PRIMARY KEY AUTOINCREMENT,  -- 自增主键
    session_id TEXT NOT NULL,          -- 关联会话ID
    role TEXT NOT NULL,               -- 角色：user/assistant
    content TEXT NOT NULL,            -- 消息内容
    created_at TEXT NOT NULL,         -- 创建时间
    FOREIGN KEY (session_id) REFERENCES sessions(id)
);
```

**字段设计说明**：

| 字段 | 类型 | 约束 | 设计原因 |
|------|------|------|----------|
| id | INTEGER | PRIMARY KEY AUTOINCREMENT | SQLite 推荐的主键类型，高效B+树索引 |
| session_id | TEXT | NOT NULL, FK→sessions | **会话隔离字段**，核心隔离机制 |
| role | TEXT | NOT NULL | 支持消息渲染区分（用户/助手） |
| content | TEXT | NOT NULL | 消息体，最大支持64KB |
| created_at | TEXT | NOT NULL | 时间线排序，消息顺序保障 |

**缺少的索引**（建议添加）：
```sql
CREATE INDEX idx_messages_session_id ON messages(session_id);
CREATE INDEX idx_messages_created_at ON messages(created_at);
CREATE INDEX idx_messages_session_time ON messages(session_id, created_at);
```

---

### 2.3 记忆表 (memories)

```sql
CREATE TABLE memories (
    id TEXT PRIMARY KEY,           -- UUID，记忆唯一标识
    type TEXT NOT NULL,         -- 记忆类型：操作习惯/分析结论/关键决策
    content TEXT NOT NULL,       -- 记忆内容
    importance INTEGER DEFAULT 1,  -- 重要性 1-5
    session_id TEXT,          -- 关联会话ID（可选，为空表示共享记忆）
    created_at TEXT NOT NULL,  -- 创建时间
    vector_id TEXT          -- 向量库ID（ChromaDB）
);
```

**字段设计说明**：

| 字段 | 类型 | 约束 | 设计原因 |
|------|------|------|----------|
| id | TEXT | PRIMARY KEY | 与向量库ID对应 |
| type | TEXT | NOT NULL | 记忆分类，便于检索/展示 |
| content | TEXT | NOT NULL | 记忆内容 |
| importance | INTEGER | DEFAULT 1 | 重要性分级，影响检索排序 |
| session_id | TEXT | 可选 | **为空表示跨会话共享记忆** |
| created_at | TEXT | NOT NULL | 时间线 |
| vector_id | TEXT | 可选 | 向量库关联，便于更新/删除 |

**当前索引**：
- 主键索引：`sqlite_autoindex_memories_1`（自动生成）

## 三、索引策略

### 3.1 当前索引（问题）

当前**缺少业务索引**，仅依赖主键：
- ❌ 无 `session_id` 索引 → 查询某会话消息需全表扫描
- ❌ 无 `created_at` 索引 → 时间排序效率低

### 3.2 推荐索引（优化后）

```sql
-- 会话表索引
CREATE INDEX idx_sessions_updated_at ON sessions(updated_at DESC);

-- 消息表索引（核心）
CREATE INDEX idx_messages_session_id ON messages(session_id);
CREATE INDEX idx_messages_created_at ON messages(created_at);
CREATE INDEX idx_messages_session_time ON messages(session_id, created_at);

-- 记忆表索引（核心）
CREATE INDEX idx_memories_session_id ON memories(session_id);
CREATE INDEX idx_memories_type ON memories(type);
CREATE INDEX idx_memories_importance ON memories(importance DESC);
```

### 3.3 索引性能对比

| 查询场景 | 无索引 | 有索引 | 提升 |
|---------|-------|-------|------|
| 查询某会话的所有消息 | O(n) 全表 | O(log n) | 10x+ |
| 按时间排序消息 | O(n log n) | O(log n) | 显著 |
| 批量会话列表 | O(n) | O(log n) | 显著 |

## 四、存储方式对比

### 4.1 方案A：共用数据表（当前实现）

```
┌─────────────────────────────────────────────────────┐
│                  messages 表                     │
├─────────┬───────────────┬────────────────────────────┤
│ session_id (隔离字段)                           │
├─────────┼─────────────┤                      │
│ 会话A   │ 会话B      │ 会话C...              │
│ msg1    │ msg1       │ msg1                  │
│ msg2    │ msg2       │ ...                  │
└─────────┴─────────────┴──────────────────────┘
```

**优点**：
- ✅ 扩展性强：支持任意数量会话
- ✅ 事务一致：跨会话操作原子性
- ✅ 维护简单：单一表便于管理
- ✅ 备份方便：一个文件导出

**缺点**：
- ❌ 数据量大时查询需带 WHERE 条件
- ❌ 需手动维护 session_id 索引

**适用场景**：
- 会话数量 < 1000
- 单用户/小团队使用
- 需要跨会话共享记忆

---

### 4.2 方案B：独立数据表

```
┌─────────────────┐  ┌─────────────────┐
│ messages_A 表    │  │ messages_B 表    │
├─────────────────┤  ├────────────────┤
│ msg1           │  │ msg1          │
│ msg2           │  │ msg2          │
└─────────────────┘  └────────────────┘
```

**优点**：
- ✅ 查询无需WHERE：直接操作单表
- ✅ 故障隔离：一表出问题不影响其他

**缺点**：
- ❌ 扩展性差：需动态DDL
- ❌ 维护复杂：备份/迁移需遍历
- ❌ 事务不一致：跨会话操作困难

**适用场景**：
- 超大会话（>100万消息）
- 多租户隔离（严格安全要求）
- 独立沙箱环境

## 五、当前架构问题

### 5.1 缺少的索引

```sql
-- 问题现象：
-- 查询某会话消息时全表扫描，10000条消息时响应 > 200ms

-- 修复：
CREATE INDEX idx_messages_session_id ON messages(session_id);
CREATE INDEX idx_messages_created_at ON messages(created_at);
```

### 5.2 前端 localStorage 隔离

前端采用**会话级别键隔离**：
```typescript
// 每个会话独立存储
const getStorageKey = () => `tongyong_messages_${sessionId}`
```

**会话切换流程**：
```
1. 用户切换会话 → sessionId 变化
2. useEffect 检测到变化 → 立即清空 messages[]
3. 异步加载 → localStorage.getItem(`tongyong_messages_${sessionId}`)
4. 设置状态 → 渲染新会话消息
```

## 六、最佳实践建议

### 6.1 场景判断

| 场景 | 推荐方案 | 原因 |
|------|----------|------|
| 小型项目（<1000会话） | 共用表 + 索引 | 简单高效 |
| 中型项目（1000-10000） | 共用表 + 分区 | 平衡扩展和简单 |
| 大型项目（>10000） | 分库分表 | 严格隔离 |

### 6.2 性能优化清单

| 优化项 | 优先级 | 预期提升 |
|--------|--------|----------|
| 添加 session_id 索引 | P0 | 10x+ 查询 |
| 批量写入缓冲 | P1 | 5x 写入 |
| 消息分页加载 | P1 | 内存降低 |
| 历史记录压缩 | P2 | 长期性能 |

### 6.3 监控指标

| 指标 | 目标 | 告警阈值 |
|------|------|----------|
| 单会话消息查询 | <50ms | >200ms |
| 会话切换响应 | <200ms | >500ms |
| 单会话消息量 | <10000 | >50000 |

## 七、数据流完整路径

### 7.1 消息存储路径

```
用户发送消息
    ↓
1. 后端保存 → messages 表（session_id 关联）
2. 后端向量化 → ChromaDB（session_id 元数据）
3. 前端保存 → localStorage（会话级别键）
```

### 7.2 数据恢复路径

```
会话切换
    ↓
1. 前端清空 → messages[] = []
2. 加载后端 → messages 表 WHERE session_id
3. 加载前端 → localStorage.getItem(key)
4. 设置状态 → 渲染
```

## 八、总结

| 维度 | 当前实现 | 建议 |
|------|----------|------|
| **存储方案** | 共用表 | 保持，小项目无需改动 |
| **会话隔离** | session_id 字段 | 核心机制，确保正确传递 |
| **前端隔离** | 会话级 localStorage | 已修复切换污染问题 |
| **后端隔离** | session_id 过滤 | ChromaDB 搜索时传入 |
| **索引** | 缺少 | 建议添加 idx_messages_session_id |

**核心原则**：所有查询都带上 `session_id` 条件，确保会话数据严格隔离。
